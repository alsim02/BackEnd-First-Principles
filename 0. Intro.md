# Appunti: Cos’è un Backend, come funziona e perché è necessario

## 1. Cos’è un backend
- Un sistema di backend è un insieme di servizi e logiche che vivono sul lato server, accessibili tramite Internet.
- È costituito da applicazioni che ascoltano richieste HTTP, WebSocket, gRPC o altri protocolli su porte aperte (es. 80, 443).
- Il backend fornisce risposte, servizi e dati ai client, come browser, app mobili, API esterne, ecc.

## 2. Come funziona un backend
- **Requisiti principali:**
  - Ascolta richieste in modo asincrono.
  - Risponde con contenuti statici (immagini, HTML, JS, CSS) o dinamici (JSON, dati elaborati).
- **Flusso di richiesta:**
  1. Il client (browser, app) invia richiesta al dominio.
  2. La richiesta raggiunge il DNS, che risolve il nome di dominio in un indirizzo IP.
  3. La richiesta passa tramite firewall e security group (in AWS o altri cloud).
  4. Arriva al server (EC2, container, ecc.) tramite IP pubblico.
  5. Un reverse proxy (come Nginx) gestisce il routing sulla porta corretta (HTTP/HTTPS).
  6. La richiesta raggiunge la applicazione (ad esempio, una app Node.js o Go in localhost:3000).
  7. L’applicazione elabora la richiesta, interagisce con database, servizi esterni, cache, ecc.
  8. La risposta viene restituita al client attraverso tutte le tappe.

## 3. Perché abbiamo bisogno di backend
- Lato client (front end), le capacità di elaborazione sono limitate:
  - Non possono accedere direttamente a database o risorse sensibili.
  - Le API del browser sono restrittive (sandboxing, CORS).
  - Non possono mantenere connessioni persistenti con database.
  - Hanno capacità computazionale limitata.
- **Esempio pratico:**
  - Attraverso il backend si gestiscono operazioni di:
    - Salvataggio dati (come i like di Instagram).
    - Notifiche (push, in-app).
    - Interazione con database e servizi esterni (pagamenti, autenticazioni).
  
## 4. Architettura del sistema
- Il backend agisce come un *nodo centrale* che:
  - Riceve richieste dai client.
  - Interagisce con database, API esterne, cache.
  - Gestisce regole di business e logica applicativa.
  - Assegna risorse e mantiene lo stato globale del sistema.

## 5. Perché non fare tutto nel front end
- **Sicurezza:** accesso diretto a database o file system è impossibile dai browser.
- **Performance:** i client (smartphone, laptop, ecc.) hanno risorse limitate per elaborazioni pesanti.
- **Gestione delle connessioni:** i server possono mantenere pool di connessioni persistenti, cosa impossibile per i browser.
- **Flessibilità:** il backend può centralizzare logica complessa, sicurezza e regole di accesso, gestire l’autenticazione, autorizzazioni e risorse condivise.

## 6. Esempio pratico di richiesta dal frontend
- Browser carica l’HTML, JS e CSS.
- JS fetcha dati e invia richieste API.
- Il server backend riceve, elabora, interagisce con DB e altri servizi.
- Risponde con dati (JSON) o esegue azioni (p.es., invio email, salvataggio).

## 7. Componenti principali di un backend
- **API endpoints:** punti di accesso alle funzionalità.
- **Database:** archivia dati strutturati e relazionali.
- **Servizi esterni:** pagamento, autenticazione, email.
- **Caches:** Redis, Memcached per performance.
- **Filtri di sicurezza:** autenticazioni, autorizzazioni, rate limiting.
- **Log e monitoraggio:** per capire il funzionamento e diagnosticare problemi.

## 8. Vantaggi di avere un backend
- **Sicurezza:** Risorse sensibili e dati protetti.
- **Scalabilità:** Gestire milioni di utenti, richieste parallele.
- **Centralizzazione:** Logica di business e regole centralizzate.
- **Migliore gestione di risorse:** connessioni, sessioni, transazioni.
- **Performance:** Cache, connection pooling, bilanciamento.

## 9. Conclusioni
- Il backend è il cuore di sistemi complessi, centralizzando le operazioni, la sicurezza e il database.
- La sua funzione è imprescindibile per applicazioni moderne, distribuite e sicure.
- Capire bene come funziona è fondamentale per sviluppare sistemi resilienti, sicuri e performanti.
